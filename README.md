# Система мониторинга обращений ЖКХ и оценки ОМСУ

![Python](https://img.shields.io/badge/Python-3.12-blue)
![Django](https://img.shields.io/badge/Django-5.x-0C4B33)
![PostgreSQL](https://img.shields.io/badge/PostgreSQL-supported-336791)
![PyQt6](https://img.shields.io/badge/Desktop-PyQt6-41CD52)

Репозиторий содержит программную часть выпускной квалификационной работы по теме мониторинга сообщений граждан о ЖКХ и благоустройстве, анализа негативного сигнала в адрес органов местного самоуправления и визуализации агрегированных показателей по округам.

Система не заменяет официальную оценку эффективности ОМСУ. Ее задача - собирать открытые сообщения из социальных сетей, выделять ЖКХ-релевантные обращения, рассчитывать дополнительные аналитические признаки и передавать результаты в клиентские интерфейсы для оперативного просмотра.

## Возможности

- веб-сервис ручной разметки сообщений и контроля качества экспертной аннотации;
- импорт и подготовка корпуса сообщений для обучения и проверки моделей;
- автоматизированный мониторинг открытых источников VK без использования VK API;
- очистка, дедупликация и нормализация новых публикаций и комментариев;
- каскадная ML-классификация по ЖКХ-таксономии;
- отдельный оценочный слой негативного сигнала в адрес ОМСУ;
- расчет агрегированных показателей, графиков и последних релевантных сообщений;
- JSON API для настольного и мобильного клиентских интерфейсов;
- настольная Windows-панель на PyQt6 с картой, карточками округов, графиками и лентой комментариев;
- эксплуатационные материалы для развертывания через nginx, gunicorn, systemd, Docker и PostgreSQL.

## Архитектура

Основной рабочий контур построен как последовательность:

```text
открытые источники VK
        |
        v
сбор публикаций и комментариев
        |
        v
очистка, дедупликация, привязка к округу
        |
        v
ML-каскад классификации
        |
        v
оценочный слой ОМСУ и агрегированные срезы
        |
        v
OMSU API
        |
        v
desktop / mobile dashboard
```

Веб-сервис разметки используется для подготовки и проверки доверенного корпуса. Серверный мониторинг работает отдельно: он регулярно собирает новые сообщения, анализирует их и обновляет таблицы, из которых API отдает готовые данные клиентам.

## Структура проекта

Структура репозитория соответствует составу программного приложения, описанного в ВКР:

- `annotation/` - основной Django-модуль веб-сервиса: модели данных, формы, представления, административная логика, API, команды импорта, разметки и мониторинга.
- `deploy/` - файлы развертывания и эксплуатации: nginx, gunicorn, systemd, fail2ban, резервное копирование и вспомогательные скрипты.
- `desktop_dashboard/` - исходный код настольного Windows-приложения для визуализации карты, графиков и обращений.
- `docker/` - файлы контейнеризации и вспомогательные материалы для развертывания.
- `docs/` - проектная документация, аналитические записки, инструкции и материалы, использованные при подготовке работы.
- `experiments/` - скрипты подготовки данных, обучения моделей, построения графиков и расчета экспериментальных метрик.
- `labeling_server/` - настройки серверного Django-проекта, маршрутизация, WSGI/ASGI-конфигурация.
- `templates/` - HTML-шаблоны интерфейса веб-сервиса и служебных страниц.
- `manage.py` - основной корневой Python-файл управления Django-проектом.

Дополнительно в корне находятся файлы зависимостей и развертывания: `requirements.txt`, `monitoring_requirements.txt`, `desktop_requirements.txt`, `Dockerfile`, `docker-compose.yml`, `.env.example`.

## Быстрый запуск веб-сервиса

```powershell
python -m venv .venv
.\.venv\Scripts\python -m pip install --upgrade pip
.\.venv\Scripts\python -m pip install -r requirements.txt
Copy-Item .env.example .env
.\.venv\Scripts\python manage.py migrate
.\.venv\Scripts\python manage.py createsuperuser
.\.venv\Scripts\python manage.py runserver 127.0.0.1:8000
```

После запуска:

- веб-интерфейс: `http://127.0.0.1:8000/`;
- административная панель Django: `http://127.0.0.1:8000/admin/`;
- проверка состояния: `http://127.0.0.1:8000/healthz/`.

Для PostgreSQL укажите `DATABASE_URL` в `.env`. Если переменная не задана, Django использует локальную SQLite-базу для разработки.

## Мониторинг источников

Зависимости мониторинга вынесены отдельно, чтобы базовый Django-сервис мог запускаться без браузерных и ML-пакетов:

```powershell
.\.venv\Scripts\python -m pip install -r monitoring_requirements.txt
.\.venv\Scripts\python -m playwright install chromium
```

Пример разового запуска тестового сбора:

```powershell
.\.venv\Scripts\python manage.py collect_vk_monitoring_test `
  --lookback-minutes 120 `
  --max-sources 10 `
  --max-posts-per-source 15 `
  --max-comments-per-post 20 `
  --analysis-batch-size 8 `
  --update-dashboard `
  --json-output runtime\vk_monitoring_latest.json
```

Для регулярной эксплуатации используются systemd timer/service из каталога `deploy/systemd/`. Типовой цикл: сбор новых сообщений раз в час, обновление агрегированных срезов после завершения мониторинга, частое получение последних комментариев клиентским приложением через легкий API-эндпоинт.

## OMSU API

API отдает готовые агрегированные данные в JSON и защищается API-ключом через заголовок `X-OMSU-API-Key` или `Authorization: Bearer`.

Основные эндпоинты:

- `GET /api/v1/omsu/manifest/` - версия API, контракт графиков и политика обновления.
- `GET /api/v1/omsu/snapshot/` - полный срез для карты, карточек округов и графиков.
- `GET /api/v1/omsu/areas/<slug>/` - детальная информация по выбранному округу.
- `GET /api/v1/omsu/latest-comment/?area=<slug>&limit=1` - последние сообщения для бегущей ленты.

В `.env` для API используются параметры:

```text
OMSU_API_REQUIRE_KEY=1
OMSU_API_KEYS=your-secret-key
OMSU_API_SNAPSHOT_REFRESH_SECONDS=3600
OMSU_API_COMMENT_REFRESH_SECONDS=5
```

## Настольное приложение

Запуск PyQt6-панели:

```powershell
.\.venv\Scripts\python -m pip install -r desktop_requirements.txt
.\.venv\Scripts\python desktop_dashboard\main.py
```

По умолчанию приложение может работать от локального статического среза `desktop_dashboard/assets/static_dashboard_snapshot.json` или от встроенных демонстрационных данных. Для подключения к серверному API:

```powershell
$env:OMSU_API_BASE = "https://example.org/api/v1/omsu"
$env:OMSU_API_KEY = "your-secret-key"
$env:OMSU_DASHBOARD_USE_API = "1"
.\.venv\Scripts\python desktop_dashboard\main.py
```

Подробнее: [`desktop_dashboard/README.md`](desktop_dashboard/README.md).

## Эксперименты и метрики

Каталог `experiments/` содержит рабочие скрипты для подготовки датасетов, обучения моделей и сравнения подходов. Документация и отчеты в `docs/` фиксируют:

- структуру корпуса и итоги ручной разметки;
- таксономию ЖКХ-обращений;
- эксперименты с RuBERT и каскадной классификацией;
- сравнение с готовыми LLM-моделями;
- оценочный слой негативного сигнала в адрес ОМСУ;
- проверку API, мониторинга и клиентских интерфейсов;
- эксплуатационные и защитные меры серверного контура.

## Данные и артефакты

В репозиторий намеренно не включаются:

- `.env` и другие файлы с секретами;
- API-ключи, токены, приватные ключи и учетные данные;
- виртуальные окружения;
- локальные базы данных;
- runtime-логи и временные результаты запусков;
- сырые выгрузки социальных сетей;
- обученные веса моделей и крупные экспериментальные артефакты.

Для локальной работы используются каталоги `data/`, `exports/`, `raw/` и `runtime/`; они исключены из git.

## Производственный контур

Для серверного запуска проект рассчитан на схему:

- Django + gunicorn как приложение;
- PostgreSQL как основная база данных;
- nginx как reverse proxy и HTTPS-терминация;
- systemd для веб-сервиса, мониторинга и резервного копирования;
- fail2ban и стандартные настройки Django для базовой защиты публичного сервера.

Примеры конфигураций находятся в `deploy/`. Перед публикацией необходимо заполнить `.env`, включить HTTPS-настройки, задать `ALLOWED_HOSTS`, включить `OMSU_API_REQUIRE_KEY` и проверить резервное копирование.

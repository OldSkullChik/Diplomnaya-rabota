# Оценка работы подсистемы автоматизированного мониторинга обращений по ЖКХ

В рамках выпускной квалификационной работы была разработана и проверена подсистема автоматизированного мониторинга сообщений из открытых городских и районных сообществ. Назначение подсистемы состоит в регулярном выявлении публикаций и комментариев, относящихся к тематике ЖКХ, их автоматической классификации по разработанной таксономии, расчете оценки влияния на ОМСУ и передаче агрегированных результатов в информационную панель мониторинга.

Необходимость отдельной подсистемы мониторинга обусловлена тем, что стандартное использование VK API для данной задачи оказалось непригодным: доступ к требуемому объему данных нестабилен и зависит от ограничений платформы. Поэтому был реализован альтернативный контур сбора данных на основе Playwright/Chromium, работающий с мобильной веб-версией VK. Такой подход позволил получать новые публикации и комментарии без изменения основной архитектуры сервера и без ручной выгрузки данных.

## Архитектура решения

Подсистема мониторинга включает несколько последовательно связанных этапов:

1. Хранение перечня источников мониторинга в таблице `MonitoringSource`. Источники формируются из подготовленного списка групп, сопоставленных с муниципальными образованиями.
2. Запуск отдельного сбора данных через management-команду `collect_vk_monitoring_test`.
3. Получение публикаций и комментариев за заданный временной интервал через Playwright/Chromium.
4. Очистка текста: исправление mojibake-кодировки, удаление служебных строк VK, нормализация относительных дат, удаление счетчиков и технических элементов страницы.
5. Дедупликация записей по устойчивому хэшу источника, внешнего идентификатора и текста.
6. Анализ текста через каскад ML-моделей:
   - модель многоосевой таксономии обращений;
   - отдельная модель определения негативного сигнала для ОМСУ;
   - интерпретируемый расчет итоговой оценки ОМСУ.
7. Запись результатов в таблицу `MonitoringItem` и фиксация служебной информации о запуске в `MonitoringRun`.
8. Обновление агрегированной панели ОМСУ: `OmsuDashboardSnapshot` и `OmsuLatestComment`.
9. Регулярный запуск через `systemd timer`.

Для исключения влияния на рабочую систему испытания проводились в отдельном тестовом контуре с самостоятельным env-файлом и копией базы данных. Это позволило безопасно проверить сбор, анализ и обновление панели без риска повреждения производственных данных.

## Логика анализа обращений

Каждый найденный текст проходит через каскад классификации. На первом уровне определяется принадлежность к ЖКХ и тематическая структура обращения: тема, тональность, тип обращения, ответственная сторона, аспект работы органов власти, качество сообщения и признаки сарказма. На втором уровне отдельная модель оценивает вероятность негативного сигнала для ОМСУ.

Итоговая оценка ОМСУ рассчитывается не только по вероятности бинарной модели, но и по интерпретируемым признакам таксономии. Например, жалоба на длительный невывоз мусора с указанием бездействия администрации получает сильный отрицательный вклад по нескольким признакам: тема `waste_cleaning`, тональность `negative`, тип обращения `complaint`, ответственная сторона `local_administration`, аспект `supervision`. Для таких случаев введено правило усиления: если интерпретируемая оценка уже является сильным негативом, а бинарная модель дает достаточно высокую вероятность негативного сигнала, запись помечается как `negative_omsu` даже при недостижении строгого верхнего порога. Это позволило не терять очевидные проблемные обращения в зоне пониженной уверенности.

При обновлении панели учитываются только сообщения, классифицированные как относящиеся к ЖКХ. Нерелевантные публикации, например объявления о вакансиях или жалобы на цены в заведениях питания, сохраняются в базе для аудита и дедупликации, но не попадают в графики и ленту последних обращений. Это защищает показатели ОМСУ от загрязнения бытовым и рекламным шумом.

## Проведенные испытания

Испытания выполнялись поэтапно. Сначала был проверен сбор данных без записи в базу, затем запись найденных публикаций, после этого подключены ML-зависимости, перенесены контрольные чекпоинты моделей и включен полный каскад анализа. После проверки качества классификации была включена агрегация данных для панели ОМСУ.

Ключевые результаты испытаний:

| Этап проверки | Результат |
| --- | --- |
| Первичный dry-run на 2 источниках | Источники обработаны без ошибок, публикации найдены, критических сбоев парсинга нет |
| Запись в тестовую БД | Записи успешно сохраняются в `MonitoringItem`, служебная информация фиксируется в `MonitoringRun` |
| Подключение ML-анализа | Зависимости и чекпоинты загружены, каскад анализирует найденные записи |
| Проверка качества на примерах | ЖКХ-жалоба на мусор определена как релевантная и негативная, нерелевантные публикации отнесены к `not_jkh` |
| Обновление dashboard | Снимки ОМСУ и последние обращения обновляются только по ЖКХ-релевантным сообщениям |
| Расширенный прогон на 10 источниках | `10/10` источников обработаны успешно, найдено `32` записи, проанализировано `32`, релевантных ЖКХ-сигналов `4`, ошибок `0` |
| Проверка системного запуска | `systemd` service завершился со статусом `0/SUCCESS`, timer активирован |

В одном из контрольных прогонов на 10 источниках были получены следующие показатели:

```text
sources_total = 10
sources_ok = 10
sources_failed = 0
posts_found = 28
comments_found = 4
items_created = 32
items_analyzed = 32
errors = []
dashboard.items_relevant = 4
dashboard.latest_comments_created = 4
```

После установки `systemd timer` был выполнен ручной запуск сервиса. Он завершился успешно и сформировал актуальный JSON-отчет:

```text
sources_total = 10
sources_ok = 10
sources_failed = 0
posts_found = 12
comments_found = 4
items_created = 16
items_analyzed = 16
errors = []
dashboard.snapshots_updated = 8
dashboard.latest_comments_created = 4
dashboard.items_seen = 48
dashboard.items_analyzed = 48
dashboard.items_relevant = 8
```

Таким образом, подтверждена работоспособность полного цикла: сбор данных, очистка, дедупликация, ML-классификация, расчет оценки ОМСУ, обновление панели и регулярный запуск по расписанию.

## Оценка качества работы

Проверка на реальных сообщениях показала, что модель корректно отделяет ЖКХ-релевантные обращения от нерелевантного информационного шума. Например, публикация о невывозе мусора была классифицирована как:

```text
jkh_relevance = yes
jkh_topic = waste_cleaning
sentiment = negative
appeal_type = complaint
responsible_party = local_administration
omsu_score = -80
omsu_decision = negative_omsu
```

При этом объявления о работе, обсуждение зарплаты и жалобы на цены в столовой были отнесены к `not_jkh` и не повлияли на показатели панели. Это важно для практической эксплуатации, поскольку городские сообщества содержат большое количество разнородных публикаций, не связанных с ЖКХ и деятельностью ОМСУ.

Отдельно была проверена работа dashboard-фильтрации. Первичная версия панели могла учитывать все проанализированные сообщения, включая нерелевантные. После доработки агрегация строится только по `jkh_relevance = yes`, а в отчет запуска добавлены диагностические счетчики `items_seen`, `items_analyzed` и `items_relevant`. Это повышает прозрачность работы подсистемы и позволяет быстро отличать отсутствие ЖКХ-сигналов от ошибки фильтрации.

## Производительность и эксплуатационная оценка

В текущей реализации сбор данных выполняется последовательно. Контрольный прогон по 10 источникам с лимитом 15 публикаций на источник занял около 8 минут 29 секунд. Этого достаточно для регулярного тестового и эксплуатационного режима с ротацией источников, однако полный обход всех 240 групп в рамках одного часа в последовательном режиме является избыточно тяжелым.

Для решения этой проблемы реализована ротация источников. Если задан лимит `--max-sources`, команда выбирает не одни и те же первые группы, а источники, которые еще не проверялись или проверялись давнее всего. Благодаря этому регулярный запуск постепенно покрывает весь список мониторинга, не перегружая сервер и не создавая чрезмерную нагрузку на VK.

Для дальнейшего масштабирования возможны следующие улучшения:

- параллельная обработка нескольких источников;
- разделение источников на несколько systemd-сервисов или очередей;
- выделение отдельных профилей запуска для крупных городов и малых районов;
- сохранение состояния последнего просмотренного post id для каждого источника;
- отдельная оптимизация сбора комментариев, так как именно переходы на страницы публикаций дают основную задержку.

## Итоговый вывод

Разработанная подсистема автоматизированного мониторинга показала практическую работоспособность и может использоваться как серверный контур регулярного выявления ЖКХ-обращений. В ходе испытаний подтверждены все основные функции: получение новых публикаций и комментариев, очистка текста, запись в базу данных, дедупликация, автоматическая классификация, расчет оценки ОМСУ, обновление информационной панели и запуск по расписанию.

Система корректно отделяет обращения по ЖКХ от нерелевантных публикаций, формирует объяснимые признаки классификации и рассчитывает интегральную оценку, пригодную для отображения в панели мониторинга. Полученные результаты подтверждают, что разработанное решение может использоваться не только как исследовательский прототип, но и как основа для практического мониторинга качества реакции органов местного самоуправления на обращения жителей в сфере ЖКХ.

Главным эксплуатационным ограничением текущей версии является последовательный сбор данных через Playwright. Это ограничение не влияет на корректность работы модели и логики анализа, но требует ротационного или параллельного режима для полного покрытия большого числа источников. В рамках выполненной работы реализован ротационный запуск через `systemd timer`, что обеспечивает регулярную обработку источников и делает систему пригодной для дальнейшей опытной эксплуатации.

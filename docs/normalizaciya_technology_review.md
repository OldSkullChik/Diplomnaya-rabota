## Применяемые технологии в курсовой

Сбор и подготовка данных:

- `Selenium`, `Chrome WebDriver`, `webdriver-manager` — парсинг постов VK через браузер;
- `pandas`, `csv`, регулярные выражения — очистка, нормализация, объединение комментариев с постами;
- `rich` — консольные таблицы, прогресс-бары и диагностический вывод.

Разметка:

- `transformers.pipeline` и модель `seara/rubert-tiny2-russian-sentiment` — первичная авторазметка тональности;
- `Flask` — локальный интерфейс ручной проверки;
- `pandas`, `numpy`, `scikit-learn` — хранение, подсказки и анализ меток.

Обучение модели:

- `PyTorch`;
- Hugging Face `transformers`;
- базовая модель `cointegrated/rubert-tiny2`;
- `AutoTokenizer`, `AutoModel`;
- `AdamW`, scheduler `get_linear_schedule_with_warmup`;
- `scikit-learn` для `train_test_split`, `classification_report`, `f1_score`, `confusion_matrix`.

Архитектура модели:

- общий ruBERT-энкодер;
- три отдельные головы классификации: тональность, тип обращения, адресат;
- v1 использует только текст комментария;
- v2 использует формат `post+comment` и увеличивает `max_length` до 256.

Сервер и визуализация:

- `Flask` API/веб-интерфейс для классификации;
- `matplotlib`, `pandas`, `numpy` для графиков;
- отдельный эксперимент по сарказму: `GradientBoostingClassifier`, `joblib`, признаки на основе текста и внешней sentiment-модели.

## Метрики старых моделей

`Normalizaciya/model`:

- база: `cointegrated/rubert-tiny2`;
- вход: комментарий;
- `max_length`: 128;
- средний weighted F1: 0.8791;
- тональность F1: 0.8021;
- тип обращения F1: 0.8756;
- адресат F1: 0.9596.

`Normalizaciya/model_v2`:

- база: `cointegrated/rubert-tiny2`;
- вход: пост + комментарий;
- `max_length`: 256;
- средний weighted F1: 0.8618;
- тональность F1: 0.7755;
- тип обращения F1: 0.8823;
- адресат F1: 0.9278.

Вывод: v1 лучше по общей средней метрике, v2 чуть лучше вытягивает некоторые классы типа обращения, но хуже по тональности и адресату. Главная проблема не в архитектуре, а в качестве исходных меток: старая авторазметка и ручная проверка были ограничены старой постановкой задачи и не покрывают новую ЖКХ-ориентированную схему.

Для больших LLM видеопамяти мало, но для дипломного авторазметчика на базе компактного BERT GPU должен заметно ускорить эксперименты. Предпочтительный путь — PyTorch с CUDA; DirectML рассматривать только как запасной вариант.

## План двух тестов

### Тест 1: старые данные

Цель: проверить, насколько авторазметчик сохраняет качество на корпусе курсовой.

Эталон:

- `Normalizaciya/structured/dataset_labeled.csv`.

Проверяемые поля:

- `sentiment`;
- `appeal_type`;
- частичный маппинг `addressee` к новой логике `responsible_party`/`authority_aspect`, если он будет нужен.

Ограничение: старый корпус не содержит новой ЖКХ-разметки, поэтому нельзя честно считать точность по `jkh_relevance`, `jkh_topic`, `quality` и `sarcasm` без дополнительной ручной доразметки.

Метрики:

- accuracy;
- macro F1;
- weighted F1;
- отчет по каждому классу;
- матрица ошибок для редких классов.

### Тест 2: новые данные

Цель: проверить авторазметчик на реальной дипломной таксономии и новых данных, размеченных студентами и утвержденных проверкой.

Эталон:

- экспорт утвержденных аннотаций командой `python manage.py export_annotations data/exports/approved_annotations.csv`;
- использовать только `status=approved`, потому что отклоненные записи без исправленной целевой метки не являются готовым эталоном.

Проверяемые поля:

- `jkh_relevance`;
- `jkh_topic`;
- `authority_aspect`;
- `sentiment`;
- `appeal_type`;
- `responsible_party`;
- `sarcasm`;
- `quality`.

Метрики:

- отдельно бинарная точность по `jkh_relevance`;
- отдельно multi-class метрики по каждому полю;
- macro F1 как главный показатель для несбалансированных классов;
- weighted F1 как общий показатель качества на реальном распределении;
- анализ ошибок по редким, но важных для диплома классам ЖКХ.

Минимальный объем:

- для smoke-теста достаточно 1 000-1 500 утвержденных записей;
- для осмысленного первого результата желательно 5 000 утвержденных записей;
- для защиты и устойчивого обучения лучше целиться в 10 000+ утвержденных записей, особенно если классы ЖКХ будут несбалансированы.

## Практический вывод

Старые данные полезны как исторический baseline и проверка совместимости с курсовой. Новые данные должны стать главным эталоном диплома, потому что только они отражают текущую предметную область: ЖКХ, обращения граждан, связь с органами власти и качественную проверку с пост-контекстом.

# Черновик дипломной работы по текущему состоянию проекта

Тема: **Разработка системы мониторинга и анализа эффективности деятельности органов местного самоуправления в сфере жилищно-коммунального хозяйства и благоустройства по данным социальных сетей**

Рабочая редакция подготовлена на основе структуры файла `Gleb_krome_3.docx` и фактически реализованной части проекта в репозитории `Diplomnaya-rabota`. Текст описывает только то, что уже собрано, реализовано, развернуто и проверено на текущем этапе.

## Введение

Жилищно-коммунальное хозяйство и благоустройство относятся к тем сферам, где качество работы органов местного самоуправления и связанных с ними организаций ежедневно отражается в пользовательских обращениях. Жители обсуждают отключения воды и отопления, состояние дворов, подъездов, лифтов, уборку мусора, работу управляющих компаний, начисления, тарифы и реакцию администраций на локальные проблемы. Значительная часть таких сообщений появляется не только в официальных каналах обращений, но и в социальных сетях.

Для органов местного самоуправления социальные сети являются важным дополнительным источником информации о состоянии городской среды и реакции населения. Однако ручное наблюдение за большим количеством публикаций и комментариев затруднено: сообщения распределены по разным сообществам, написаны неформальным языком, часто содержат сарказм, эмоциональные оценки, неполный контекст и ссылки на конкретные бытовые ситуации. Поэтому для практического анализа необходима система, которая позволяет собрать данные, привести их к единому формату, организовать их разметку и подготовить качественный корпус для дальнейшей автоматической классификации.

Цель работы заключается в разработке и развертывании информационной системы подготовки и ручной проверки данных из социальных сетей для последующего анализа обращений граждан в сфере жилищно-коммунального хозяйства и благоустройства.

Для достижения цели на текущем этапе решены следующие задачи:

1. Изучен существующий курсовой проект как исходная техническая и методическая база.
2. Сформирована схема подготовки данных из файлов, полученных из социальных сетей.
3. Реализована сборка единого набора данных из разрозненных CSV-выгрузок.
4. Разработана предметная схема разметки обращений по тематике ЖКХ, тональности, типу обращения, ответственной стороне, качеству записи и признаку сарказма.
5. Реализован многопользовательский веб-сервер разметки на Django.
6. Добавлены роли пользователей, проверка ответов, начисление баллов, штрафы, таблица лидеров и административная проверка.
7. Выполнено развертывание системы на физическом сервере с публичным HTTPS-доступом.
8. Реализованы меры базовой безопасности, резервного копирования, режима обслуживания, health-check и нагрузочной проверки.
9. Загружена основная очередь комментариев с контекстом исходных постов.

Объектом исследования являются пользовательские комментарии и публикации из социальных сетей, связанные с городской средой, жилищно-коммунальным хозяйством и благоустройством.

Предметом исследования является процесс подготовки, структурирования и проверки текстовых данных для последующего анализа эффективности деятельности органов местного самоуправления в сфере ЖКХ.

Инструментальная база проекта включает язык Python, фреймворк Django, PostgreSQL, nginx, gunicorn, systemd, CSV-обработку, скрипты импорта и сборки данных, а также веб-интерфейс ручной разметки.

Практическая значимость текущего результата состоит в том, что создана рабочая инфраструктура для массовой разметки комментариев несколькими пользователями. Система позволяет выдавать записи студентам, контролировать качество разметки, проверять ответы администраторами, фиксировать историю баллов и формировать проверенный корпус данных.

## Глава 1. Проектирование системы мониторинга и подготовки данных

### 1.1. Подход к анализу обращений в сфере ЖКХ по данным социальных сетей

Социальные сети содержат большое количество сообщений, которые можно рассматривать как неформальные обращения граждан. В отличие от официальных обращений, такие сообщения часто не имеют строгой структуры, но они отражают реальные проблемы, эмоциональную реакцию жителей и локальные оценки работы городских служб.

Для сферы ЖКХ особенно важны следующие свойства таких данных:

- привязка обращения к конкретной проблеме: вода, отопление, мусор, двор, подъезд, лифт, тарифы, управляющая компания;
- наличие оценки деятельности ответственных организаций или органов власти;
- эмоциональная окраска сообщения;
- тип высказывания: жалоба, вопрос, просьба, требование, благодарность, мнение или информирование;
- наличие контекста исходного поста, без которого комментарий может быть непонятен;
- наличие сарказма или иронии, из-за которых буквальный смысл текста может отличаться от реального отношения автора.

В курсовом проекте уже была реализована автоматическая классификация пользовательских комментариев по нескольким признакам. При этом была выявлена ключевая проблема: качество автоматической модели ограничивается качеством исходных меток. Если метки получены автоматически или без учета контекста поста, модель может формально показывать хорошие метрики, но допускать систематические ошибки в спорных случаях. Поэтому в дипломной версии проекта центральным этапом стала не немедленная тренировка модели, а создание инструмента для качественной ручной разметки данных с видимым контекстом поста.

Текущая версия системы ориентирована на подготовку проверенного корпуса обращений по ЖКХ. Такой корпус нужен для дальнейшего анализа и может использоваться как основа для обучения моделей классификации, построения статистики по темам и оценки реакции жителей на работу ответственных структур.

### 1.2. Проектирование информационной системы

Информационная система построена как веб-приложение с централизованной базой данных. Пользователи работают с ней через браузер, а все записи, ответы, проверки и баллы сохраняются в PostgreSQL.

Общая архитектура включает следующие компоненты:

| Компонент | Назначение |
| --- | --- |
| Django-приложение `labeling_server` | Основная серверная логика, маршруты, шаблоны, формы, авторизация |
| Приложение `annotation` | Модели данных, разметка, проверка, импорт, служебные команды |
| PostgreSQL | Хранение пользователей, исходных записей, разметок и баллов |
| nginx | Публичный HTTPS-прокси и отдача статических файлов |
| gunicorn | WSGI-сервер для запуска Django в production-режиме |
| systemd | Автозапуск gunicorn, таймеры резервного копирования и health-check |
| CSV-команды | Сборка, импорт и экспорт данных |

В системе выделены три основных уровня доступа:

1. Студент. Получает записи из очереди, размечает их и отправляет ответы на проверку.
2. Администратор разметки. Утверждает студентов, проверяет ответы, принимает или отклоняет разметку, начисляет баллы и штрафы.
3. Суперадминистратор проекта. Имеет расширенный доступ к полному списку участников, корректировке баллов и административным функциям. В текущей конфигурации этот уровень жестко привязан к учетной записи `oldskull`, чтобы случайный Django-superuser не получал проектные права автоматически.

Основные сценарии работы:

| Сценарий | Описание |
| --- | --- |
| Регистрация | Пользователь создает учетную запись и ожидает утверждения |
| Утверждение | Администратор допускает студента к работе |
| Выдача записи | Система случайно выбирает активную запись, которую пользователь еще не размечал |
| Бронирование | Запись резервируется за пользователем на 15 минут, чтобы не выдаваться одновременно другим |
| Разметка | Пользователь видит текст поста, комментарий и заполняет поля разметки |
| Отправка на проверку | Ответ сохраняется в статусе ожидания проверки |
| Проверка | Администратор принимает ответ, отклоняет его или подтверждает отметку о удаленном посте |
| Начисление | За принятый обычный ответ начисляется +1, за брак обычной разметки -2, за неверную отметку "пост удален" -1 |
| Сводная статистика | На главной странице отображается топ участников по баллам |

Внутренняя модель данных реализована через четыре ключевые сущности:

| Модель | Назначение |
| --- | --- |
| `UserProfile` | Расширение стандартного пользователя Django: роль, статус утверждения, публичное имя |
| `SourceRecord` | Исходная запись для разметки: комментарий, контекст поста, источник, активность, бронь |
| `Annotation` | Ответ пользователя по всем полям разметки и статус проверки |
| `ScoreEvent` | История начислений и штрафов с указанием администратора и причины |

Для предотвращения одновременной разметки одной записи несколькими студентами в `SourceRecord` добавлены поля `reserved_by` и `reserved_until`. При открытии новой записи система резервирует ее на 15 минут. Если пользователь не отправил ответ, бронь истекает и запись может быть выдана другому участнику.

Выдача записи реализована случайным образом среди доступных кандидатов. При этом исключаются записи, уже размеченные данным студентом, а также записи, по которым уже есть отправленный или утвержденный ответ. Отклоненные записи могут вернуться в очередь и быть размечены другим участником.

### 1.3. Безопасность, устойчивость и условия эксплуатации

Система развернута как публичный веб-сервис, поэтому при проектировании учитывались вопросы безопасности доступа, устойчивости и восстановления после сбоев.

На уровне приложения реализованы:

- регистрация с обязательным утверждением учетной записи;
- разделение ролей студента, администратора разметки и суперадминистратора;
- защита форм средствами CSRF Django;
- валидация паролей Django;
- аудит проверки через хранение проверяющего пользователя;
- хранение истории начислений и штрафов;
- отключение `DEBUG` в production-режиме;
- настройка `ALLOWED_HOSTS` и `CSRF_TRUSTED_ORIGINS`;
- HTTPS-only cookies и перенаправление на HTTPS.

На уровне сервера реализованы:

- работа Django только через gunicorn на локальном адресе `127.0.0.1:8000`;
- публичный доступ через nginx;
- HTTPS-сертификат Let's Encrypt для домена `label.zhkh-razmetka.ru`;
- ограничение открытых портов через UFW;
- установка fail2ban для защиты от повторяющихся неуспешных попыток доступа;
- ежедневное резервное копирование PostgreSQL через systemd timer;
- режим технического обслуживания;
- health-check endpoint `/healthz/`;
- статическая fallback-страница nginx на случай недоступности gunicorn.

Для операторов и студентов, работающих с системой разметки, важна эргономика интерфейса. Поэтому экран разметки сделан двухколоночным: слева отображается контекст поста и комментарий, справа находится форма решения. Форма закрепляется сбоку, чтобы пользователь мог размечать длинные тексты без постоянной прокрутки к полям ответа. Для каждого поля добавлены подсказки с примерами, открываемые кнопкой `?`.

## Глава 2. Подготовка и анализ исходных данных

### 2.1. Сбор и объединение информации из социальных сетей

Исходные данные проекта хранятся в локальной папке `raw/`. Она не коммитится в Git, так как содержит крупные выгрузки, промежуточные данные и файлы парсинга. Основным форматом является схема, полученная из проекта `Normalizaciya`:

```text
text, post_text, data_type, has_post_context, group_id, group_name, post_id,
comment_id, comment_url, date, author, likes, sentiment, appeal_type,
addressee, file_origin
```

Ключевые поля:

- `text` - основной текст для разметки;
- `post_text` - контекст исходного поста;
- `data_type` - тип строки: комментарий, пост или другой тип записи;
- `comment_url`, `post_id`, `comment_id` - идентификаторы источника;
- `group_name` - название сообщества;
- `file_origin` - файл, из которого пришла строка.

В проекте используются несколько типов исходных файлов:

1. `dataset.csv` - компактированный набор данных из курсового проекта.
2. `posts_parsed.csv` - отдельно восстановленные тексты постов для старых комментариев.
3. CSV-выгрузки комментариев Barkov.
4. CSV-выгрузки постов Barkov.

Для объединения этих файлов реализована команда:

```bash
python manage.py build_dataset raw \
  --output data/processed/dataset_combined.csv \
  --missing-output data/processed/missing_post_context.csv
```

Команда выполняет следующие действия:

- читает старый структурированный набор `dataset.csv`;
- использует `posts_parsed.csv` для восстановления контекста старых комментариев;
- читает Barkov-выгрузки постов и комментариев;
- извлекает VK post id из ссылок вида `wall-...`;
- связывает комментарии с исходными постами;
- приводит все строки к единому набору колонок;
- удаляет дубли внутри итогового файла;
- формирует отдельный файл кандидатов без найденного контекста поста.

Итоговый локальный файл `dataset_combined.csv` содержит 413 670 строк. Распределение по типам в текущей версии:

| Тип строки | Контекст поста | Количество |
| --- | ---: | ---: |
| comment | есть | 265 130 |
| comment | нет | 4 336 |
| post | нет | 142 528 |
| wallpost | есть | 1 676 |

Таким образом, основной массив для ручной разметки составляют комментарии с восстановленным или найденным контекстом исходного поста. На сервер была загружена именно эта рабочая очередь, без отдельных постов как самостоятельных заданий. После догрузки база разметчика содержит 265 181 исходную запись, из них 265 131 запись имеет контекст поста.

Для безопасной загрузки реализована команда импорта:

```bash
python manage.py import_records data/processed/dataset_combined.csv \
  --source-name combined-comments-with-posts \
  --data-type comment \
  --require-post-context \
  --batch-size 5000
```

Ключи `--data-type comment` и `--require-post-context` позволяют загрузить только комментарии, у которых есть текст исходного поста. Это исключает попадание в очередь отдельных постов и комментариев без контекста. Повторная загрузка безопасна: дубли отсеиваются через SHA-256 хэш, построенный из внешнего идентификатора, текста комментария и текста поста.

### 2.2. Схема разметки обращений

Для подготовки корпуса разработана предметная схема разметки, ориентированная на ЖКХ и благоустройство. Она учитывает не только эмоциональную окраску текста, но и связь с конкретной сферой городской среды, тип обращения и предполагаемую ответственную сторону.

Поля разметки:

| Поле | Назначение |
| --- | --- |
| Относится к ЖКХ | Определяет, связан ли текст с ЖКХ или благоустройством |
| Тема ЖКХ | Выделяет основную тему: отопление, вода, мусор, двор, подъезд, тарифы и др. |
| Работа органов/служб | Фиксирует аспект оценки работы ответственных структур |
| Тональность | Определяет эмоциональную окраску сообщения |
| Тип обращения | Жалоба, вопрос, просьба, требование, предложение, благодарность, мнение, информирование |
| Ответственная сторона | УК/ТСЖ, ресурсоснабжающая организация, администрация, жилинспекция, жители и др. |
| Сарказм или ирония | Отмечает наличие иронического смысла |
| Качество записи | Обычная, сложная, спам, дубль, недостаточно контекста |
| Комментарий к разметке | Необязательное пояснение студента |

Темы ЖКХ в текущей таксономии:

- отопление и горячая вода;
- холодная вода и канализация;
- мусор и уборка;
- подъезд, лифт, крыша, подвал;
- двор и придомовая территория;
- тарифы, квитанции, счетчики;
- работа УК/ТСЖ;
- работа администрации и государственных органов;
- другое ЖКХ;
- не ЖКХ.

Для уменьшения типовых ошибок в форме реализовано правило: если пользователь выбирает "не относится к ЖКХ", то зависимые поля автоматически нормализуются. Тема становится "Не ЖКХ", аспект работы органов/служб - "Не применимо", ответственная сторона - "Не применимо". Это правило реализовано одновременно на frontend-уровне и на backend-уровне, поэтому оно сохраняется даже при обходе интерфейса.

На практике это правило было введено после анализа первых 216 непроверенных разметок. В них были найдены случаи, когда участники выбирали "не относится к ЖКХ", но оставляли в зависимых полях ЖКХ-тему или ответственную сторону. Так как ошибка возникла из-за недостаточно строгой формы, старые записи этого типа были нормализованы без штрафа: 89 таких ответов были приняты и получили +1 балл.

### 2.3. Текущее состояние разметочного корпуса

На текущем этапе сформирована и загружена основная очередь данных для ручной проверки. Рабочая база содержит комментарии с контекстом поста, что позволяет разметчику учитывать ситуацию, к которой относится ответ пользователя.

Основные текущие показатели:

| Показатель | Значение |
| --- | ---: |
| Всего строк в `dataset_combined.csv` | 413 670 |
| Комментариев с контекстом поста в файле | 265 130 |
| Комментариев без контекста поста в файле | 4 336 |
| Отдельных постов в файле | 142 528 |
| Загружено исходных записей в базу разметчика | 265 181 |
| Записей в базе с контекстом поста | 265 131 |

Отдельные посты не загружались как задания на ручную разметку, потому что основной единицей текущего корпуса является комментарий с видимым контекстом исходной публикации. Такой формат соответствует задаче дальнейшего анализа обращений граждан: комментарий фиксирует реакцию пользователя, а пост помогает понять ситуацию и корректно определить тему, тональность и адресата.

## Глава 3. Реализация информационной системы

### 3.1. Реализация серверной части и интерфейса

Информационная система реализована как Django-проект. Основной проект называется `labeling_server`, а доменное приложение - `annotation`.

Реализованные маршруты:

| URL | Назначение |
| --- | --- |
| `/` | Главная страница и сводная статистика |
| `/signup/` | Регистрация пользователя |
| `/login/` | Вход |
| `/logout/` | Выход |
| `/pending/` | Страница ожидания утверждения |
| `/annotate/` | Получение следующей записи |
| `/annotate/<id>/` | Разметка конкретной записи |
| `/review/` | Очередь проверки |
| `/review/<id>/` | Детальная проверка ответа |
| `/review/<id>/quick/` | Быстрая проверка из очереди |
| `/participants/` | Управление участниками |
| `/leaderboard/` | Полная таблица лидеров для суперадминистратора |
| `/healthz/` | Проверка здоровья приложения и базы |
| `/admin/` | Стандартная Django-админка |

Главная страница показывает личную статистику пользователя и общий рейтинг. В рейтинге отображаются имя, фамилия или публичное имя, количество баллов, количество разметок и текущий стрик правильных ответов. Для обычных пользователей показывается топ-10. Для суперадминистратора доступен полный список и ручная корректировка баллов.

Экран разметки состоит из двух основных областей:

1. Контекстная область. Показывает ID записи, сообщество, источник, текст исходного поста и текст комментария.
2. Область решения. Содержит поля разметки и кнопки отправки.

В форме разметки доступны три основных действия:

- отправить ответ на проверку;
- отметить, что пост удален;
- пропустить запись.

Кнопка "Пост удален" нужна для случаев, когда источник нельзя проверить или контекст отсутствует из-за удаления публикации. Такая отметка отправляется администраторам на отдельную проверку. Если администратор подтверждает удаление, запись исключается из очереди без начисления баллов. Если отметка неверна, студент получает штраф -1, а запись возвращается в очередь.

Очередь проверки показывает администратору список ожидающих ответов. Для ускорения работы прямо в списке отображается краткая информация по каждой разметке: студент, фрагмент комментария, фрагмент поста, выбранные категории и служебные признаки. Администратор может быстро принять или отклонить ответ без перехода в детальную карточку.

Логика начислений:

| Событие | Результат |
| --- | ---: |
| Обычная разметка принята | +1 |
| Обычная разметка отклонена как брак | -2 |
| Отметка "пост удален" подтверждена | 0 |
| Отметка "пост удален" отклонена | -1 |
| Ручная корректировка суперадминистратора | произвольное значение |

Для всех начислений используется модель `ScoreEvent`, поэтому история баллов не теряется и может быть проверена.

### 3.2. Пример работы системы

Работа с системой начинается с регистрации пользователя. После регистрации пользователь попадает на страницу ожидания утверждения. До утверждения он не может получать записи для разметки.

Администратор разметки открывает список участников и утверждает учетную запись студента. После этого студент получает доступ к главной странице и кнопке начала разметки.

При переходе к разметке система выбирает случайную активную запись из очереди и резервирует ее за студентом на 15 минут. Если другой студент в этот момент запросит новую запись, зарезервированная запись ему не попадет. Это снижает вероятность параллельной работы нескольких участников над одним и тем же комментарием.

Студент читает пост и комментарий, затем выбирает значения по полям разметки. Если комментарий не относится к ЖКХ, часть зависимых полей заполняется автоматически. Если пост недоступен, студент может выбрать действие "Пост удален".

После отправки ответ получает статус "ожидает проверки". Администратор открывает очередь проверки, видит краткую сводку по ответам и принимает решение. При принятии ответ становится утвержденным, а студент получает балл. При отклонении запись возвращается в очередь, чтобы ее мог разметить другой участник.

Суперадминистратор видит полную таблицу участников, может просматривать общий рейтинг и вносить ручные корректировки баллов. Это используется для исправления служебных ситуаций, когда ошибка возникла из-за изменения правил системы, а не из-за действий студента.

### 3.3. Развертывание и проверка работоспособности

Система развернута на физическом сервере под Ubuntu 24.04 LTS. Сервер использует процессор AMD A10-7860K Radeon R7 и около 8 ГБ оперативной памяти. Приложение работает в CPU-режиме и не требует GPU на текущем этапе, так как основной функционал связан с веб-интерфейсом, базой данных и обработкой CSV.

Текущая схема deployment:

```text
Пользователь
    -> HTTPS
    -> nginx
    -> gunicorn 127.0.0.1:8000
    -> Django
    -> PostgreSQL
```

Публичный домен: `https://label.zhkh-razmetka.ru`.

Выполненные шаги развертывания:

- установлен PostgreSQL;
- создана база данных проекта;
- применены Django-миграции;
- создан центральный superuser;
- настроен gunicorn как systemd service;
- настроен nginx как reverse proxy;
- включен HTTPS через Let's Encrypt;
- настроены production security flags через `.env`;
- включена отдача статических файлов;
- настроен UFW;
- установлен fail2ban;
- настроены автоматические резервные копии PostgreSQL;
- настроены maintenance mode и health endpoint;
- выполнены нагрузочные проверки.

В production-конфигурации gunicorn запускается с 4 синхронными workers. Такое значение было выбрано после сравнения производительности. При 2 workers login-page stress test показывал около 644 запросов в секунду. После увеличения до 4 workers throughput вырос примерно до 1064 запросов в секунду, а средняя задержка на высоких уровнях concurrency снизилась примерно на 39%. При этом память gunicorn выросла примерно до 165 МБ, что приемлемо для сервера с доступными 6,7 GiB RAM.

Для проверки доступности реализован endpoint:

```json
{"status": "ok", "database": "ok", "maintenance": false}
```

Он проверяет доступность Django и соединение с базой данных. Endpoint остается доступным даже во время режима обслуживания.

Результаты нагрузочных проверок:

| Проверка | Результат |
| --- | --- |
| `/healthz/`, до 600 запросов и concurrency 20 | 0 failed requests |
| `/login/`, до 30 000 запросов и concurrency 150 | 0 failed requests |
| `/login/`, 225 000 запросов при concurrency 200/300/400 на 2 workers | 0 failed requests, около 644 req/s |
| `/login/`, 225 000 запросов при concurrency 200/300/400 на 4 workers | 0 failed requests, около 1064 req/s |

После нагрузочных проверок приложение оставалось в состоянии `active`, endpoint `/healthz/` возвращал `ok`, swap не использовался. Это подтверждает достаточную устойчивость текущей серверной конфигурации для задачи ручной разметки.

Для обслуживания реализован специальный режим технических работ. Он включается командой управления Django и возвращает пользователям страницу `503 Service Unavailable` с поясняющим сообщением. Также реализован вариант с таймером обратного отсчета. Это позволяет временно закрывать интерфейс для пользователей во время исправления ошибок или обновления данных.

Резервное копирование реализовано через systemd timer. Скрипт создает сжатые дампы PostgreSQL в директории `/home/oldskull/backups/diplom` и удаляет устаревшие копии старше 14 дней.

Тестирование показало, что текущая конфигурация с 4 workers обеспечивает устойчивую работу и достаточный запас производительности для ручной разметки.

```bash
  --output data/processed/dataset_combined.csv \
```

```bash
python manage.py import_records data/processed/dataset_combined.csv \

  --data-type comment \

  --batch-size 5000
```

```bash
python manage.py export_annotations exports/approved_annotations.csv
```

```bash
curl https://label.zhkh-razmetka.ru/healthz/
sudo systemctl status diplom-gunicorn --no-pager
```

```bash
python manage.py maintenance on \
  --title "Вскоре" \
  --message "Сервис временно переведен в режим обслуживания." \

```

Отключение режима обслуживания:

```bash
python manage.py maintenance off
```

# Анализ просадки отдельных осей классификации

Дата анализа: 2026-06-03.

Источник метрик:

```text
data\ml_experiments\teacher_student_runs\sweep_final_2026-06-03\final_w03_weighted_lr1e5_e4\metrics.json
data\exports\project_advanced_charts_2026-06-03\confusion_matrices.json
```

Проверка выполнялась на отложенной human-gold test-выборке из `655` записей.
Эта выборка не содержит automatic teacher/silver-разметку, поэтому метрики отражают
качество модели именно относительно проверенных человеком данных.

## Как читать метрики

В анализе используются три основные метрики:

- `accuracy` - доля полностью верных предсказаний по конкретной оси;
- `weighted F1` - F1 с учетом количества примеров в каждом классе;
- `macro-F1` - средний F1 по всем классам, где каждый класс имеет одинаковый
  вес независимо от того, сколько примеров этого класса есть в тесте.

Именно `macro-F1` используется как ключевая аналитическая метрика для этой
задачи, потому что данные сильно несбалансированы. Если в оси есть один
доминирующий класс, модель может получать высокую `accuracy`, почти всегда
выбирая этот класс, но плохо распознавать редкие классы. В этом случае
`weighted F1` будет выглядеть приемлемо, а `macro-F1` останется низким.

Это не противоречие в метриках, а признак дисбаланса. Для дипломной задачи это
важно: модель должна не только отличать самые частые случаи, но и сохранять
качество на редких категориях, связанных с ответственными сторонами, работой
власти, типами обращений и качеством комментариев.

## Общая причина просадки

Четыре наиболее слабые оси:

```text
responsible_party
authority_aspect
appeal_type
quality
```

На русском:

- `responsible_party` - ответственная сторона;
- `authority_aspect` - аспект работы власти или ответственных организаций;
- `appeal_type` - тип обращения или форма реакции;
- `quality` - качество комментария для разметки.

Основная причина низкого `macro-F1` по этим осям - не общий отказ модели, а
структура данных:

1. В тестовой выборке преобладают доминирующие классы.
2. Редкие классы представлены единицами или несколькими десятками примеров.
3. Некоторые категории смыслово пересекаются в живых комментариях.
4. Часть осей зависит не только от текста, но и от интерпретации ситуации:
   кто именно должен отвечать, является ли реплика просьбой или требованием,
   относится ли оценка к власти, управляющей компании или просто к обсуждению.

## Почему accuracy выглядит лучше, чем macro-F1

Для осей `responsible_party`, `authority_aspect` и `quality` accuracy выглядит
сравнительно высокой, потому что модель хорошо угадывает доминирующий класс:

| Ось | Доминирующий класс | Доля в test |
| --- | --- | ---: |
| Ответственная сторона | Не применимо | 90.2% |
| Аспект работы власти | Не применимо | 90.8% |
| Качество | Нормальный комментарий | 87.8% |

Если модель часто выбирает доминирующий класс, она получает много правильных
ответов по количеству строк. Но `macro-F1` усредняет качество по классам, а не
по строкам. Поэтому редкие классы с нулевым или низким F1 резко снижают итог.

Пример: для `responsible_party` класс `не применимо` имеет F1 `85.9%`, но сразу
пять редких классов имеют F1 `0%`. В результате средний `macro-F1` равен только
`14.8%`.

## Что это означает для дальнейшей работы

Общее увеличение объема данных само по себе не решит проблему. Если добавить
еще много простых записей класса `не применимо` или `нормальный комментарий`,
accuracy может вырасти, но `macro-F1` по редким классам почти не изменится.

Нужен целевой добор данных:

1. Для `responsible_party` - больше примеров, где явно различаются:
   управляющая компания, ресурсоснабжающая организация, администрация,
   жилищная инспекция, оператор ТКО и жители.
2. Для `authority_aspect` - больше примеров бездействия, плохого качества,
   медленной реакции, надзора, тарифной политики и коммуникации.
3. Для `appeal_type` - отдельные примеры на границах:
   мнение / жалоба / вопрос / просьба / требование / предложение.
4. Для `quality` - больше размеченных примеров спама, дублей и сложных случаев.

Наиболее полезный следующий набор для обучения - не случайная выборка, а
`hard-gold`: записи, где текущая модель не уверена, путает редкие классы или
расходится с правилами. Такой набор будет меньше по размеру, но даст больше
прироста качества, чем большое количество легких однотипных примеров.

## Формулировка для отчета

Результаты показывают, что модель успешно усваивает общую структуру задачи и
лучше всего работает на частых классах, однако качество классификации редких
управленческих и интерпретационных категорий остается ограниченным. Основная
причина заключается в дисбалансе обучающих и тестовых данных, а также в
семантической близости некоторых классов. Поэтому дальнейшее повышение качества
требует не только увеличения общего числа размеченных записей, но и
целенаправленного пополнения выборки редкими и пограничными примерами.

# Читаемый аналитический свод ML-метрик проекта

Дата фиксации: `2026-06-06`.

Этот файл является человекочитаемой версией сводного аналитического отчета по ML-части дипломного проекта. Он сохраняет ключевые числа, но объясняет их проще: что именно измерялось, почему часть результатов оказалась низкой, какие выводы можно честно использовать в ВКР, а какие лучше показывать только как исследовательские попытки.

Главное правило чтения: проверка качества модели считается только на `human-gold` данных. `silver` и `pseudo-gold` можно использовать для обучения, расширения корпуса и аналитики, но они не являются независимой экспертной истиной.

## 1. Что именно мы получили

Проект построен вокруг анализа обращений и общественных реакций в сфере ЖКХ. На входе есть пост, комментарий и восстановленный контекст обсуждения. На выходе система должна давать:

- подробную 8-осевую таксономию для статистики и инфографики;
- отдельную числовую оценку работы ОМСУ;
- вероятность негативного сигнала в адрес ОМСУ;
- решение по уверенности модели: можно ли доверять выводу автоматически или лучше отправить на проверку.

Итоговая честная картина такая:

| Направление | Итог |
| --- | --- |
| Полная original 8-осевая таксономия | Работает как исследовательская аналитика, но текущий потолок качества около `0.29-0.33` mean macro-F1 |
| Grouped taxonomy | Дает лучший результат за счет укрупнения классов: до `0.3561` test mean macro-F1 |
| Большие модели RuBERT | Не улучшили результат относительно оптимизированного `rubert-tiny2` |
| Весовые эксперименты | Не дали существенного роста для original taxonomy |
| Pseudo-gold | Полезен точечно, но не заменяет human-gold |
| OMSU selective layer | Сильный практический слой: coverage `0.7496`, macro-F1 `0.9102`, negative F1 `0.8235` |

Самый важный вывод для диплома: низкий F1 полной 8-осевой таксономии не отменяет применимость системы. Детальная таксономия слишком сложная и несбалансированная для имеющегося объема human-gold, зато отдельная задача оценки ОМСУ решается заметно надежнее.

## 3. Как устроены данные

В проекте есть три типа разметки. Их нельзя смешивать при интерпретации качества.

| Тип | Кто/что разметил | Для чего использовать | Можно ли считать test/validation |
| --- | --- | --- | --- |
| `human-gold` | Человек, проверенная разметка | Обучение, validation, test, дипломные метрики | Да |
| `silver` | Автоматическая/teacher-student разметка | Расширение train, предварительная авторазметка, анализ покрытия | Нет |
| `pseudo-gold` | Строго отобранная teacher-assisted разметка | Точечное усиление слабых классов в train | Нет |

Почему это важно: если проверять модель на silver, получится проверка модели против похожего автоматического источника, а не против человека. Такая оценка может выглядеть красиво, но для диплома она не доказывает реальное качество модели.

## 6. Почему качество нельзя оценивать только accuracy

В данных очень сильный дисбаланс. Например, ранний accepted snapshot показал:

| Класс | Количество | Доля |
| --- | ---: | ---: |
| Положительные ЖКХ-записи | `351` | `7.90%` |
| Не-ЖКХ | `4073` | `91.69%` |
| Неуверенные | `18` | `0.41%` |
| Всего training-ready на тот момент | `4442` | `100%` |

Если модель почти всегда говорит “не ЖКХ”, accuracy может быть высокой, но модель будет плохо ловить главный для диплома класс. Поэтому основной показатель для сложной таксономии - `macro-F1`, а не accuracy.

`Macro-F1` считает качество по каждому классу отдельно, а потом усредняет. Поэтому редкие классы не исчезают за счет массовых классов.

`Weighted-F1` и accuracy полезны как дополнительные показатели, но их нельзя делать главными: они растут, когда модель хорошо угадывает частые классы вроде `not_jkh`, `not_applicable`, `normal`.

## 9. Восемь осей таксономии

Полная таксономия не является одной простой классификацией. Это восемь разных задач одновременно.

| Ось | Русский смысл | Что определяет |
| --- | --- | --- |
| `jkh_relevance` | ЖКХ-релевантность | Относится ли запись к сфере ЖКХ |
| `jkh_topic` | Тема ЖКХ | Отопление, мусор, двор, тарифы и т.д. |
| `authority_aspect` | Аспект работы власти | Бездействие, качество работы, контроль, тарифная политика |
| `sentiment` | Тональность | Позитивная, негативная, нейтральная реакция |
| `appeal_type` | Тип обращения | Жалоба, требование, вопрос, благодарность, предложение |
| `responsible_party` | Ответственная сторона | Администрация, УК, ресурсник, оператор ТКО, жители |
| `sarcasm` | Сарказм | Есть ли саркастический/ироничный тон |
| `quality` | Качество текста | Нормальный текст, шум, дубль, нет контекста |

Пост и комментарий в этой схеме играют разные роли:

| Элемент | Роль |
| --- | --- |
| Пост | Задает тему, контекст и ЖКХ-релевантность |
| Комментарий | Показывает реакцию общества: тональность, сарказм, тип обращения, детали проблемы |
| Ветка/контекст | Помогает понять короткие и ироничные реакции |

Именно поэтому в проекте было принято правило: направление темы определяет пост, а комментарий уточняет реакцию.

## 11. Логическая постобработка original taxonomy

После модели применяются простые правила согласованности. Главное правило:

```text
если jkh_relevance = no,
то:
  jkh_topic = not_jkh
  authority_aspect = not_applicable
  responsible_party = not_applicable
```

Это не “новая модель”, а логическая защита от противоречий. Если запись не про ЖКХ, у нее не должно быть ЖКХ-темы или ответственного ЖКХ-субъекта.

| Показатель | Значение |
| --- | ---: |
| Baseline до правил | `0.3003` mean macro-F1 |
| После consistency rules | `0.3032` mean macro-F1 |
| Строгая all-label оценка | `0.2920` mean macro-F1 |
| Strict all-8 exact match | `0.0885` |
| Затронуто test-записей | `36/655` |

## 13. Grouped taxonomy

Grouped taxonomy - это укрупненная версия таксономии. Она не является прямой заменой original taxonomy, потому что классы объединены. Но она полезна как практическая альтернатива, когда новой human-gold разметки уже не будет.

### Что группировалось

| Ось | Как укрупнялось |

Selective taxonomy - это режим, где модель отвечает только тогда, когда достаточно уверена. Чем жестче порог уверенности, тем меньше coverage, но иногда выше качество на оставшейся части.

## 20. Числовая оценка ОМСУ

Пользовательская задача не ограничивается тональностью. Нужно получить число, которое описывает направленность оценки работы ОМСУ. Поэтому был выделен отдельный OMSU score layer.

Он не заменяет 8 осей. Он работает поверх них и рядом с отдельным бинарным классификатором негативного сигнала.

### Поля OMSU-слоя

| Поле | Что означает |
| --- | --- |
| `omsu_score` | Числовая оценка от `-100` до `100` |
| `omsu_negative_probability` | Вероятность негативного сигнала в адрес ОМСУ |
| `omsu_decision` | Итоговое решение: негативно / не негативно / низкая уверенность |
| `omsu_confidence_band` | Уровень уверенности |
| `omsu_score_reason` | Краткое объяснение, почему выставлена такая оценка |

### Формула простыми русскими словами

```text
оценка_ОМСУ =
  округлить(
    ограничить(
      базовая_оценка * вес_связи_с_ОМСУ * вес_качества,
      от -100 до 100
    )
  )

базовая_оценка =
  баллы_тональности
  + баллы_типа_обращения
  + баллы_аспекта_работы_власти

вес_связи_с_ОМСУ =
  максимум(
    вес_ответственной_стороны,
    вес_темы_ЖКХ * 0.85,
    вес_аспекта_работы_власти
  )
```

Если запись не относится к ЖКХ или имеет качество `spam/duplicate`, итоговая оценка принудительно равна `0`. Это нужно, чтобы не выставлять оценку ОМСУ по мусорным или нерелевантным данным.

### Почему множители такие

| Множитель | Зачем нужен |
| --- | --- |
| `баллы_тональности` | Отражают эмоциональный знак реакции: жалоба и негатив должны тянуть оценку вниз |
| `баллы_типа_обращения` | Требование или жалоба сильнее обычного нейтрального комментария |
| `баллы_аспекта_работы_власти` | Позволяют отличать плохую работу, бездействие, контроль, благодарность |
| `вес_ответственной_стороны` | Усиливает оценку, если в тексте явно фигурирует администрация, ОМСУ или другой публичный субъект |
| `вес_темы_ЖКХ * 0.85` | Тема ЖКХ важна, но сама по себе не всегда является оценкой ОМСУ, поэтому вес немного ниже прямой ответственности |
| `вес_качества` | Снижает влияние плохих, шумных или неполных текстов |
| Ограничение `-100..100` | Делает шкалу стабильной для API, графиков и сравнения муниципалитетов |

## 22. Почему для ОМСУ понадобилась отдельная модель

Сначала была проверена производная оценка: взять предсказанные 8 осей и вычислить `omsu_score` только из них.

| Подход | Validation macro-F1 | Test macro-F1 | Test weighted-F1 | Test MAE score |
| --- | ---: | ---: | ---: | ---: |
| Производная оценка от 8 осей | `0.6129` | `0.6371` | `0.8502` | `13.14` |

Результат неплохой, но недостаточный для уверенной рейтинговой оценки. Ошибки слабых осей переносятся в итоговую цифру.

Поэтому был обучен отдельный бинарный классификатор `negative_omsu`.

| Параметр | Значение |
| --- | --- |
| Модель | `cointegrated/rubert-tiny2` |
| Задача | `negative_omsu` vs `not_negative_omsu` |
| Text mode | `post_comment` |
| Max length | `192` |
| Batch size | `16` |
| Learning rate | `1e-5` |
| Epochs | `3` |
| Train rows | `20000` |
| Class weights | `weighted_balanced`, power `0.75`, cap `4` |
| Silver weight | `0.3` |

| Selective `p>=0.85 / p<=0.15` | `0.9939` | `0.9102` | `0.9937` | `0.8235` |

## 23. Финальная архитектура системы

Финальная схема не сводит все к одной модели. Это каскад.

| Шаг | Что происходит | Зачем нужно |
| --- | --- | --- |
| 1 | На вход подается пост + комментарий | Пост задает тему, комментарий задает реакцию |
| 2 | Модель предсказывает 8 осей | Получаем подробную таксономию для статистики |
| 3 | Применяются consistency rules | Убираются логические противоречия |
| 4 | Отдельная модель определяет `negative_omsu` | Оценка ОМСУ не зависит полностью от слабых осей |
| 5 | Рассчитываются `omsu_score`, probability, decision, confidence band, reason | Получаем числовой рейтинговый слой |
| 6 | Результаты уходят в API, приложение и инфографику | Используется в прикладной системе |

Таким образом:

- 8 осей нужны для подробной аналитики;
- OMSU layer нужен для числовой оценки работы ОМСУ;
- selective confidence нужен, чтобы не выдавать сомнительные выводы как надежные.

| No sample weights | `0.2962` | Нет | Silver нельзя уравнивать с gold | Обосновать sample weights |
| Individual class weights | `0.3000` | Нет | Не превзошло baseline | Показать как отрицательный результат |
| Pseudo-gold | Full `0.2385`, hybrid около `0.3069` | Частично | Полезен точечно, широкое применение шумит | Использовать аккуратно |
| Selective taxonomy | До `0.3363`, coverage `0.5076` | Частично | Уверенная часть лучше, но покрытие падает | Для confidence-политики |
| OMSU binary layer | Macro-F1 `0.9102`, coverage `0.7496` | Да | Отдельная задача проще и ближе к целевому рейтингу | Главный прикладной ML-результат |

| ЖКХ-internal specialist | Routed `0.2864` | Нет | Улучшения внутри ЖКХ не спасли полный каскад | Отклоненный финальный тест |

## 25. Ограничения исследования

Эти ограничения лучше прямо проговорить в дипломе. Они делают выводы честнее.

| Ограничение | Как влияет |
| --- | --- |
| Малый human-gold для редких классов | Модель плохо учит редкие категории |
| Сильный дисбаланс классов | Accuracy и weighted-F1 могут завышать качество |
| Шум silver | Silver помогает масштабу, но может переносить ошибки teacher-разметки |
| Неоднозначные границы таксономии | Некоторые классы трудно отличить даже человеку |
| Сарказм и короткие комментарии | Реакция часто понятна только из контекста |
| Grouped taxonomy укрупняет классы | Ее нельзя напрямую сравнивать с original taxonomy |
| Pseudo-gold не экспертная разметка | Нельзя использовать для validation/test |
| OMSU selective покрывает не все строки | Высокое качество достигается на уверенной части |

## 27. Что говорить на защите простыми словами

| Вопрос | Короткий ответ |
| --- | --- |
| Почему F1 по 8 осям невысокий? | Потому что задача слишком детальная, а часть классов встречается в gold всего несколько раз. |
| Это провал? | Нет. Это честное ограничение полной таксономии; при этом отдельный слой оценки ОМСУ дал сильный результат. |
| Почему не помогла большая модель? | Большая модель не создает новые качественные примеры редких классов. Проблема больше в данных, чем в размере encoder. |
| Зачем grouped taxonomy? | Она показывает, что при укрупнении классов качество растет, значит часть проблемы в слишком тонких границах классов. |
| Зачем OMSU-модель отдельно? | Потому что числовая оценка ОМСУ - отдельная прикладная задача, и ее лучше решать напрямую, а не только через слабые оси. |
| Почему selective-режим честный? | Система не притворяется уверенной везде: уверенные случаи идут в автоматическую аналитику, сомнительные помечаются отдельно. |
| Что нужно для роста до 85% по полной таксономии? | Больше human-gold по редким классам, упрощение классов или более строгая экспертная схема разметки. |

## 28. Финальный вывод

За последние эксперименты была проверена почти вся разумная зона улучшений без новой ручной разметки: большие модели, веса классов, веса выборки, grouped taxonomy, pseudo-gold, selective policies, specialists и финальные ансамбли.

Главный честный результат:

- полная original 8-осевая таксономия остается на уровне примерно `0.29-0.33` mean macro-F1;
- grouped taxonomy дает более устойчивую, но укрупненную альтернативу до `0.3561`;
- отдельный OMSU selective layer дает сильный прикладной результат: `0.9102` macro-F1 при coverage `0.7496`;
- финальная система должна использовать каскад, а не одну модель “на все”.

Для диплома это хорошая история: проект не просто обучил модель, а показал границы применимости сложной таксономии, обосновал выбор метрик, построил полный корпус и выделил отдельный надежный слой для оценки работы ОМСУ.
